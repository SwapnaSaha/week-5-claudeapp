# Auth Spec

## Feature Name
Signup and Login (Custom Auth)

## Description
Email and password authentication using a custom `users` table — no Supabase Auth. Passwords hashed with `bcryptjs` at 10 rounds. On successful auth, `userId` and `userEmail` are stored in `localStorage`. Users are redirected to `/dashboard`. Both pages are standalone light-mode cards (no app shell).

---

## User Flow — Signup

1. User navigates to `/signup`
2. User fills in: email, password, confirm password
3. Client validates: all fields non-empty, password and confirm password match
4. Client calls `POST /api/auth/signup` with `{ email, password }`
5. Server checks if email already exists in `users` table → 409 if found
6. Server hashes password with bcryptjs (10 rounds)
7. Server inserts new row into `users` table
8. Server returns `{ id, email }` with status 201
9. Client stores `userId` and `userEmail` in `localStorage`
10. Client redirects to `/dashboard`

---

## User Flow — Login

1. User navigates to `/login`
2. User fills in: email, password
3. Client validates: both fields non-empty
4. Client calls `POST /api/auth/login` with `{ email, password }`
5. Server queries `users` table by email
6. Server compares submitted password with stored hash using `bcryptjs.compare`
7. Any failure (email not found or password mismatch) → 401 with generic message
8. Server returns `{ id, email }` with status 200
9. Client stores `userId` and `userEmail` in `localStorage`
10. Client redirects to `/dashboard`

---

## DB Schema

See `specs/database.md` — `users` table.

Key points:
- `email` is UNIQUE — enforced at database level
- `password_hash` stores the bcrypt output (never the raw password)
- `id` is a UUID generated by `gen_random_uuid()`

## API Routes

### `POST /api/auth/signup`
**Request body:** `{ email: string, password: string }`

| Status | Body | Condition |
|---|---|---|
| 201 | `{ id, email }` | User created successfully |
| 409 | `{ error: "Email already in use" }` | Email already exists in users table |
| 400 | `{ error: "Email and password required" }` | Missing fields |
| 500 | `{ error: "Server error" }` | Unexpected failure |

### `POST /api/auth/login`
**Request body:** `{ email: string, password: string }`

| Status | Body | Condition |
|---|---|---|
| 200 | `{ id, email }` | Credentials valid |
| 401 | `{ error: "Invalid email or password" }` | Email not found OR password mismatch (same message — never reveal which) |
| 400 | `{ error: "Email and password required" }` | Missing fields |
| 500 | `{ error: "Server error" }` | Unexpected failure |

### `GET /api/auth/logout`
- Deletes the `azure_token` HTTP-only cookie
- Client clears `userId` and `userEmail` from `localStorage`
- Redirects to `/login`

## Auth Guard

Every protected page (`/dashboard`, `/chat/[sessionId]`) runs this check on mount:

```ts
useEffect(() => {
  const userId = localStorage.getItem('userId')
  if (!userId) router.replace('/login')
}, [])
```

localStorage key: `userId` (string UUID)
localStorage key: `userEmail` (string)

---

## Important Implementation Notes
- Store exactly `userId` and `userEmail` as keys in localStorage
- Login error must be generic — never reveal whether email or password was wrong
- Use `bcryptjs` (not native `bcrypt`) — it works in Next.js API routes without native bindings
- Signup must check email existence before hashing (fail fast)
- Logout must delete the `azure_token` HTTP-only cookie in addition to clearing localStorage

## Edge Cases

| Scenario | Handling |
|---|---|
| Email already registered (signup) | 409 → "Email already in use" shown below form |
| Passwords don't match (signup) | Client-side block before API call, inline error |
| Wrong password (login) | 401 → "Invalid email or password" (generic) |
| Email not found (login) | 401 → same generic message |
| Empty field submission | Client-side block + API-level 400 |
| Network error | Show "Something went wrong. Please try again." |
| Already logged in (visits /login) | Redirect to /dashboard if `localStorage.userId` exists |
